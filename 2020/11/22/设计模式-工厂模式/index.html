前言

最近项目中，一直在负责对接外部资金(银行),功能模块几乎一样，只是对应银行的接口字段有所差别。项目初期，使用银行编码区分，使用ifelse判断具体走哪个方法。前期项目较少，所以ok。后来随着接入的资金方越来越多，防止后期代码修改，影响所有，以及代码的优雅性，改成了Map<String code,Fuction<String,Result>>接口，对应不同的方法。现在改造成 工厂模式，对代码进行重构。

工厂模式简介

工厂模式又称工厂方法模式，是一种创建型设计模式，其在父类中提供一个创建对象的方法， 允许子类决定实例化对象的类型。
这种设计模式也是 Java 开发中最常见的一种模式，它的主要意图是定义一个创建对象的接口，让其子类自己决定实例化哪一个工厂类，工厂模式使其创建过程延迟到子类进行。
简单说就是为了提供代码结构的扩展性，屏蔽每一个功能类中的具体实现逻辑。让外部可以更加简单的只是知道调用即可，同时，这也是去掉众多ifelse的方式。当然这可能也有一些缺点，比如需要实现的类非常多，如何去维护，怎样减低开发成本。但这些问题都可以在后续的设计模式结合使用中，逐步降低。

总结

工厂模式的优点: 避免创建者与具体的产品逻辑耦合，满足单一职责，每一个具体的业务都在自己所属的类中完成，满足开闭原则，无需更改调用的方法就可以在程序中引入新的业务类型。问题:后期产品增多，实现的子类会急速扩张。
